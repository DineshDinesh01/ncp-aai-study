1. Slurm Architecture

Slurm (Simple Linux Utility for Resource Management) is the dominant workload scheduler in HPC and AI clusters. It manages job queuing, resource allocation (GPUs, CPUs, memory), and node state tracking.

ComponentRoleRuns On
slurmctldCentral controller daemon — maintains cluster state, schedules jobsHead/login node
slurmdWorker daemon — executes jobs, reports node stateEvery compute node (DGX)
slurmdbdAccounting database daemon — records job history, usageDatabase server
sinfoShow cluster partitions and node stateCLI (any node)
squeueShow running and pending jobsCLI
sbatchSubmit batch jobsCLI
scontrolAdmin: drain, resume, show configCLI (admin)

2. sinfo — Cluster State

# Basic cluster state view
sinfo

# Output columns:
# PARTITION  AVAIL  TIMELIMIT  NODES  STATE  NODELIST
# dgx        up     7-00:00:0   4     idle   dgx[01-04]

# Show all nodes with detailed state
sinfo -N -l

# Show GPU resources available
sinfo -o "%N %G %C %t"       # Node, GRES, CPU info, state
sinfo --Format=nodelist,gres,cpus,memory,statecompact

# Show nodes by state
sinfo -t idle                 # Only idle nodes
sinfo -t drain                # Drained nodes (removed from scheduling)
sinfo -t down                 # Down nodes

# Node state codes:
# idle   = available, no jobs running
# alloc  = fully allocated
# mix    = partially allocated
# drain  = draining (admin marked, no new jobs)
# down   = node unreachable
# resv   = in reservation

3. squeue — Job Queue

# Show all jobs
squeue

# Show jobs for specific user
squeue -u username

# Show jobs on specific partition
squeue -p dgx

# Detailed job info with GPU allocation
squeue -o "%i %j %u %t %R %b"    # JobID, Name, User, State, Reason, GRES

# Watch queue live
watch -n 5 squeue

# Job state codes:
# R  = Running
# PD = Pending (waiting for resources)
# CG = Completing
# CA = Cancelled
# F  = Failed

# Pending reasons:
# Resources        = waiting for free GPUs/CPUs
# Priority         = lower priority than other pending jobs
# QOSMaxJobsPerUser = user job limit reached
# Dependency       = waiting for another job

4. sbatch — Job Submission

#!/bin/bash
#SBATCH --job-name=llm-train
#SBATCH --partition=dgx
#SBATCH --nodes=2                    # 2 DGX nodes
#SBATCH --ntasks-per-node=8         # 8 tasks per node (1 per GPU)
#SBATCH --gpus-per-node=8           # 8 GPUs per node
#SBATCH --gres=gpu:8                # GRES GPU request
#SBATCH --cpus-per-task=12          # CPU threads per task
#SBATCH --mem=0                     # All available memory
#SBATCH --time=24:00:00             # 24-hour time limit
#SBATCH --output=logs/job-%j.out    # %j = job ID
#SBATCH --error=logs/job-%j.err

# Load modules
module load cuda/12.3
module load openmpi/4.1.5

# Run with Pyxis/Enroot container
srun --container-image=nvcr.io/nvidia/pytorch:24.01-py3 \
     --container-mounts=/data:/data \
     python train.py --nodes 2 --gpus 16
# Submit the job
sbatch train_job.sh

# Submit with inline options (override script)
sbatch --nodes=4 --partition=h100 train_job.sh

# Job arrays (sweep over hyperparameters)
sbatch --array=1-10 train_job.sh   # Creates jobs 1-10, $SLURM_ARRAY_TASK_ID set

# Cancel a job
scancel <jobid>

# Cancel all your jobs
scancel -u $USER

5. scontrol — Admin Control

# Show detailed job information
scontrol show job <jobid>

# Show node detailed info
scontrol show node dgx01

# Show partition config
scontrol show partition dgx

# Drain a node (stop new job scheduling, wait for running jobs to finish)
scontrol update NodeName=dgx01 State=DRAIN Reason="GPU maintenance"

# Resume a drained/down node
scontrol update NodeName=dgx01 State=RESUME

# Set a node to DOWN
scontrol update NodeName=dgx01 State=DOWN Reason="Hardware failure"

# Hold a pending job (prevent it from starting)
scontrol hold <jobid>

# Release a held job
scontrol release <jobid>

# Requeue a running job (restart it)
scontrol requeue <jobid>

6. GPU GRES Configuration

Generic Resource Scheduling (GRES) is how Slurm tracks GPU resources. Configuration is in /etc/slurm/gres.conf on each node:

# /etc/slurm/gres.conf on each DGX node
Name=gpu Type=h100 File=/dev/nvidia0
Name=gpu Type=h100 File=/dev/nvidia1
Name=gpu Type=h100 File=/dev/nvidia2
Name=gpu Type=h100 File=/dev/nvidia3
Name=gpu Type=h100 File=/dev/nvidia4
Name=gpu Type=h100 File=/dev/nvidia5
Name=gpu Type=h100 File=/dev/nvidia6
Name=gpu Type=h100 File=/dev/nvidia7

# /etc/slurm/slurm.conf (cluster config, relevant lines)
GresTypes=gpu
NodeName=dgx01 Gres=gpu:h100:8 CPUs=128 RealMemory=2048000
# Request specific GPU type in job
sbatch --gres=gpu:h100:4 myjob.sh    # 4× H100 GPUs

# Request any GPU type
sbatch --gpus-per-node=8 myjob.sh

# Check GRES allocation in running job
scontrol show job <jobid> | grep GRES
MIG + Slurm GRES
With MIG enabled, each MIG instance is a separate GRES entry. Configure gres.conf with MIG device files and use --gres=gpu:1g.10gb:4 to request specific MIG profiles in Slurm jobs.

7. Draining & Resuming Nodes

Before maintenance (driver update, GPU replacement, firmware update), drain the node:

# 1. Drain the node (gracefully stop new allocations)
scontrol update NodeName=dgx01 State=DRAIN Reason="Replacing GPU 4"

# 2. Wait for running jobs to complete
squeue -w dgx01               # Watch until no jobs remain

# 3. Perform maintenance (GPU swap, driver update, etc.)

# 4. Verify system is healthy
nvidia-smi -L                 # All GPUs visible?
dcgmi diag -g 1 -r 2          # Run DCGM diagnostics

# 5. Resume the node
scontrol update NodeName=dgx01 State=RESUME

# 6. Verify node is back in Slurm
sinfo -N | grep dgx01         # Should show 'idle'

8. Exam Prep & Quick Reference

Must-Know for Exam (19% domain weight — Control Plane)
  • Slurm daemons: slurmctld (controller) and slurmd (worker on each node)
  • GPU GRES configured in /etc/slurm/gres.conf and referenced in slurm.conf
  • Drain a node before maintenance: scontrol update NodeName=... State=DRAIN
  • Resume after maintenance: State=RESUME
  • sinfo -t drain shows nodes in maintenance mode
  • Pending reason "Resources" = not enough free GPUs
  • scontrol show job <id> gives full job details including GRES allocation
  • Node state: idle → alloc (fully used) → mix (partially used)
