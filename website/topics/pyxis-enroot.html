Pyxis & Enroot

NVIDIA's rootless container stack for HPC clusters — Enroot for image management and Pyxis for Slurm integration, replacing Docker in multi-tenant GPU environments.

1. Why Not Docker in HPC?

Docker requires root daemon access, which is a security risk in shared HPC environments. Traditional HPC clusters use many users sharing the same nodes — running a root daemon violates the principle of least privilege.

PropertyDockerEnroot + Pyxis
Requires root daemonYesNo (rootless)
Multi-tenant safeNoYes
Slurm integrationManual / complexNative via Pyxis plugin
NGC image supportManual pull+tagDirect nvcr.io:// URI
Container isolationFull daemon isolationUser-level namespaces
MPI supportComplexBuilt-in (PMIx)

2. Enroot — Container Runtime

Enroot is NVIDIA's lightweight, rootless container runtime that converts OCI/Docker images into squashfs files and runs them as unprivileged user processes.

# Import a Docker/OCI image into Enroot
enroot import docker://ubuntu:22.04
enroot import docker://nvcr.io/nvidia/pytorch:24.01-py3

# Import from NGC with authentication
enroot import docker://nvcr.io#nvidia/pytorch:24.01-py3

# List imported containers
enroot list

# Create a container instance from imported image
enroot create --name pytorch-env nvidia+pytorch+24.01-py3.sqsh

# Start an interactive container
enroot start --root --rw pytorch-env

# Start with specific mount
enroot start --mount /data:/data --mount /scratch:/scratch pytorch-env

# Remove a container
enroot remove pytorch-env
Enroot Image Format
Enroot stores images as .sqsh (squashfs) files. The naming convention follows the registry path with + separating components: nvcr.io#nvidia/pytorch:24.01-py3 becomes nvidia+pytorch+24.01-py3.sqsh.

3. Pyxis — Slurm Plugin

Pyxis is a Slurm SPANK plugin that allows users to run Enroot containers directly from srun and sbatch without admin intervention.

# Verify Pyxis is installed (Slurm plugin)
ls /usr/lib/slurm/spank_pyxis.so

# Pyxis configuration: /etc/slurm/plugstack.conf
# required /usr/lib/slurm/spank_pyxis.so

# Run a container with srun
srun --container-image=nvcr.io/nvidia/pytorch:24.01-py3 \
     --gpus-per-node=8 \
     python3 train.py

# Run with local sqsh file
srun --container-image=/shared/images/pytorch.sqsh \
     --gpus-per-node=8 \
     python3 train.py

4. Pulling NGC Images

NGC (NVIDIA GPU Cloud) is NVIDIA's registry for pre-built AI/ML containers, optimized for DGX hardware.

# NGC API key authentication for private containers
# Set in environment or ~/.config/enroot/credentials.conf

# Download with NGC CLI
ngc registry image pull nvcr.io/nvidia/pytorch:24.01-py3

# Pull directly with Enroot (requires NGC API key)
export ENROOT_LOGIN_EXTRA="Bearer $NGC_API_KEY"
enroot import docker://nvcr.io/nvidia/pytorch:24.01-py3

# Common NGC containers for AI:
# nvcr.io/nvidia/pytorch:24.01-py3          — PyTorch + CUDA + cuDNN
# nvcr.io/nvidia/tensorflow:24.01-tf2-py3   — TensorFlow 2
# nvcr.io/nvidia/nemo:24.01                 — NeMo for LLM training
# nvcr.io/nvidia/tritonserver:24.01-py3     — Triton Inference Server
# nvcr.io/nvidia/cuda:12.3.0-devel-ubuntu22.04 — Base CUDA dev image

5. srun Container Flags (Pyxis)

# Key Pyxis flags for srun/sbatch:

# --container-image    : Image URI or path to .sqsh file
# --container-mounts   : Mount directories into container
# --container-workdir  : Working directory inside container
# --container-name     : Named container (persists between steps)
# --container-save     : Save container state on exit
# --no-container-remap-root : Run as root inside container (debugging)

# Full example — multi-node distributed training with Pyxis
#SBATCH --nodes=4
#SBATCH --gpus-per-node=8

srun \
  --container-image=nvcr.io/nvidia/pytorch:24.01-py3 \
  --container-mounts=/lustre/data:/data,/lustre/checkpoints:/checkpoints \
  --container-workdir=/workspace \
  python3 -m torch.distributed.launch \
    --nproc_per_node=8 \
    --nnodes=4 \
    --node_rank=$SLURM_NODEID \
    train_llm.py

6. Container Mounts & Storage

Containers are ephemeral — data must be mounted from the host file system. DGX clusters use parallel file systems:

Storage TypeTypical MountUse Case
Lustre (parallel FS)/lustreTraining datasets, checkpoints — high-bandwidth parallel I/O
NFS/home, /sharedUser home dirs, small shared files
Local NVMe/scratch or /tmpFast local cache, staging datasets
NGC Model StoreDownloaded on demandPre-trained model weights
 
# Mount multiple directories
srun --container-image=pytorch:24.01-py3 \
     --container-mounts=/lustre/datasets:/data,/lustre/output:/output,/home/$USER:/home/$USER \
     python3 train.py

# Check Lustre mount is healthy
df -h /lustre
lfs df /lustre             # Lustre-specific space report
lfs find /lustre/data -type f -name "*.tar" | wc -l

7. Exam Prep & Quick Reference

Must-Know for Exam (19% Control Plane domain)
  • Enroot = rootless container runtime for HPC (replaces Docker daemon)
  • Pyxis = Slurm SPANK plugin enabling Enroot containers in srun/sbatch
  • Docker not used in multi-tenant HPC — requires root daemon which is a security risk
  • enroot import docker://nvcr.io/... pulls and converts NGC images to .sqsh
  • srun --container-image= flag is the Pyxis way to run containers in jobs
  • Pyxis configured via /etc/slurm/plugstack.conf → required spank_pyxis.so
  • Container data must be mounted — containers are ephemeral
  • Lustre provides high-bandwidth parallel I/O for training datasets
