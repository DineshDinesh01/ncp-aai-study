1 What is LangGraph?

LangGraph is an open-source library built on top of LangChain that lets you construct stateful, cyclic, multi-actor agent workflows as a directed graph. Released in early 2024, it addresses the fundamental limitation of linear LangChain chains: you cannot loop back, branch conditionally, or persist state across turns without custom glue code.

The three primitives of LangGraph are:

  • StateGraph — the graph object that holds your typed shared state
  • Nodes — Python functions (or runnables) that read from and write to state
  • Edges — deterministic or conditional connections between nodes
Key insight: Every node receives the full current state as input and returns a partial update (a dict). LangGraph merges that update back into the state automatically using reducer functions — the most common being operator.add for lists (append semantics).

LangGraph is used by teams building production agents at companies including Elastic, Replit, and LinkedIn. It is the recommended orchestration layer when you need human-in-the-loop interrupts, multi-turn memory, and complex branching logic that would be difficult to express as a linear chain.

2 Why LangGraph Exists

Standard LLM API calls are stateless — each call is independent. Building a real agent requires:

  • Remembering what happened in previous steps
  • Looping back to retry or refine outputs
  • Routing to different nodes depending on the LLM's decision
  • Pausing for human approval mid-execution
  • Persisting the agent's progress across sessions (fault tolerance)

Before LangGraph, developers hacked together these patterns with Python classes, Redis, and custom state machines. LangGraph provides a principled, graph-native abstraction that encodes all of these patterns directly.

Exam angle: LangGraph enables cycles (loops) in agent graphs. Standard DAG-based frameworks like Prefect or Airflow do not support cycles. This is the differentiator — an agent that loops until a condition is met requires a cyclic graph.

3.1 Defining State

State is a TypedDict with type annotations. Fields with Annotated[list, operator.add] use append semantics — each node adds to the list rather than replacing it.

python state_definition.py 
from typing import TypedDict, Annotated, List
import operator

class AgentState(TypedDict):
    messages: Annotated[List[str], operator.add]   # append on each update
    current_step: str                               # replace on each update
    iteration_count: int                            # replace on each update
    final_answer: str                               # replace on each update

3.2 Nodes

A node is any callable that accepts the current state dict and returns a partial update dict. Only the keys you include in the return dict are updated — other fields remain unchanged.

python nodes.py 
def agent_node(state: AgentState) -> dict:
    # Reads from state, returns only what changes
    response = llm.invoke(state["messages"])
    return {
        "messages": [response.content],   # appended (Annotated list)
        "current_step": "agent_done"       # replaced
    }

def tool_node(state: AgentState) -> dict:
    last_message = state["messages"][-1]
    result = execute_tool(last_message)
    return {
        "messages": [f"Tool result: {result}"],
        "current_step": "tool_done"
    }

3.3 Edges and Conditional Routing

Regular edges always go from node A to node B. Conditional edges call a routing function that returns the name of the next node.

python routing.py 
from langgraph.graph import StateGraph, START, END

def should_use_tool(state: AgentState) -> str:
    """Router: returns the name of the next node."""
    last_msg = state["messages"][-1]
    if "TOOL_CALL:" in last_msg:
        return "tool_node"
    return END  # string "end" also accepted

# Build graph
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("tool_node", tool_node)

workflow.add_edge(START, "agent")          # deterministic edge
workflow.add_conditional_edges(            # conditional routing
    "agent",
    should_use_tool,
    {"tool_node": "tool_node", END: END}   # mapping: return_value -> node_name
)
workflow.add_edge("tool_node", "agent")    # loop back to agent

3.4 Checkpointing

Checkpointing saves the full state after every node execution. It enables:

  • Human-in-the-loop — pause at a node, wait for user input, then resume
  • Fault tolerance — restart from the last checkpoint if a node crashes
  • Multi-turn memory — same thread_id resumes the prior conversation
python checkpointing.py 
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer)

# thread_id groups messages into a single conversation session
config = {"configurable": {"thread_id": "user-session-42"}}

result1 = app.invoke({"messages": ["Hello!"]}, config=config)
result2 = app.invoke({"messages": ["What did I just say?"]}, config=config)
# result2 has full history — state was persisted between calls

5 Full Python Example — ReAct Agent with LangGraph

This example builds a complete ReAct-style agent using LangGraph. It includes a tool node, an agent node, conditional routing, and MemorySaver checkpointing.

python langgraph_react_agent.py 
"""
Full LangGraph ReAct Agent Example
- StateGraph with typed state
- Tool node + Agent node
- Conditional routing
- MemorySaver checkpointing
"""

import operator
from typing import TypedDict, Annotated, List, Optional
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage, ToolMessage
from langchain_core.tools import tool
import json


# ── 1. DEFINE TOOLS ──────────────────────────────────────────────────────────

@tool
def search_web(query: str) -> str:
    """Search the web for information."""
    # In production, call a real search API here
    return f"[Search result for '{query}']: NVIDIA H100 GPU has 80GB HBM3 memory."

@tool
def calculate(expression: str) -> str:
    """Evaluate a mathematical expression."""
    try:
        result = eval(expression, {"__builtins__": {}})
        return str(result)
    except Exception as e:
        return f"Error: {e}"

TOOLS = [search_web, calculate]
TOOLS_BY_NAME = {t.name: t for t in TOOLS}


# ── 2. DEFINE STATE ───────────────────────────────────────────────────────────

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]   # full message history (append)
    iteration: int                             # loop counter (replace)
    final_answer: Optional[str]               # set when done (replace)


# ── 3. DEFINE NODES ───────────────────────────────────────────────────────────

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
llm_with_tools = llm.bind_tools(TOOLS)

def agent_node(state: AgentState) -> dict:
    """Agent node: calls the LLM with the full message history."""
    response = llm_with_tools.invoke(state["messages"])
    return {
        "messages": [response],
        "iteration": state.get("iteration", 0) + 1,
    }

def tool_node(state: AgentState) -> dict:
    """Tool node: executes all tool calls in the last AI message."""
    last_message = state["messages"][-1]
    tool_results = []

    for tool_call in last_message.tool_calls:
        tool_fn = TOOLS_BY_NAME[tool_call["name"]]
        result = tool_fn.invoke(tool_call["args"])
        tool_results.append(
            ToolMessage(
                content=result,
                tool_call_id=tool_call["id"]
            )
        )
    return {"messages": tool_results}


# ── 4. ROUTING FUNCTION ───────────────────────────────────────────────────────

def should_continue(state: AgentState) -> str:
    """
    Route after the agent node:
    - If the last message has tool calls → go to tool_node
    - If iteration limit hit → END
    - Otherwise → END (agent gave a final answer)
    """
    last_message = state["messages"][-1]
    MAX_ITERATIONS = 5

    if state.get("iteration", 0) >= MAX_ITERATIONS:
        return END

    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tool_node"

    return END


# ── 5. BUILD GRAPH ────────────────────────────────────────────────────────────

workflow = StateGraph(AgentState)

# Add nodes
workflow.add_node("agent", agent_node)
workflow.add_node("tool_node", tool_node)

# Entry point
workflow.add_edge(START, "agent")

# Conditional routing from agent
workflow.add_conditional_edges(
    "agent",
    should_continue,
    {
        "tool_node": "tool_node",
        END: END,
    }
)

# After tool execution, always go back to agent
workflow.add_edge("tool_node", "agent")

# Compile with checkpointing
checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer)


# ── 6. INVOKE ─────────────────────────────────────────────────────────────────

if __name__ == "__main__":
    config = {"configurable": {"thread_id": "demo-session-1"}}

    # Turn 1
    result = app.invoke(
        {"messages": [HumanMessage(content="What GPU does NVIDIA's H100 use?")],
         "iteration": 0},
        config=config
    )
    print("Final messages:")
    for msg in result["messages"]:
        role = type(msg).__name__
        content = getattr(msg, "content", "")
        print(f"  [{role}]: {content[:120]}")

    # Turn 2 — state is resumed from checkpoint
    result2 = app.invoke(
        {"messages": [HumanMessage(content="What is 1024 * 80?")]},
        config=config
    )
    print("\nTurn 2 final message:")
    print(" ", result2["messages"][-1].content)

6 NVIDIA NIM as the LLM Backend
NVIDIA NIMs

NVIDIA NIMs are containerized, optimized LLM inference microservices. They expose an OpenAI-compatible REST API, so swapping from OpenAI to a NIM requires changing only the base_url and model.

python langgraph_with_nim.py 
from langchain_openai import ChatOpenAI

# Drop-in replacement: point to your local NIM endpoint
llm = ChatOpenAI(
    base_url="http://localhost:8000/v1",   # NIM container port
    api_key="not-required",                # NIMs don't need a key locally
    model="meta/llama-3.1-70b-instruct",   # NIM model name
    temperature=0,
    max_tokens=1024,
)

# Bind tools exactly as before
llm_with_tools = llm.bind_tools(TOOLS)

# Everything else in the LangGraph code is IDENTICAL
# The graph structure, state, nodes, edges — all unchanged

For production deployments on NVIDIA DGX or cloud (AWS, GCP, Azure), NVIDIA NIMs can be launched with a single docker run. LangGraph scales naturally because it separates orchestration logic from the inference backend.

Exam point: NVIDIA NIMs expose the same API schema as OpenAI's Chat Completions API (including tool_calls in responses), so tool binding with llm.bind_tools() works identically. No code changes needed in your LangGraph nodes.
7 Lab — Code Review Agent with Conditional Loop

Build a "code review" agent with 3 nodes and a conditional loop that allows up to 3 review iterations before forcing an exit.

python code_review_agent.py 
"""
Lab: Code Review Agent
  Node 1: write_code   — LLM writes initial code
  Node 2: review_code  — LLM reviews and scores the code (1-10)
  Node 3: fix_code     — LLM fixes issues found in review

Conditional loop: if score < 7 and iterations < 3, loop back to fix_code.
"""

import operator
from typing import TypedDict, Annotated, List, Optional
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)

# ── STATE ─────────────────────────────────────────────────────────────────────

class CodeReviewState(TypedDict):
    task_description: str
    code: str
    review_feedback: Annotated[List[str], operator.add]  # accumulate all feedback
    review_score: int
    iteration: int
    final_code: Optional[str]

# ── NODE 1: WRITE CODE ────────────────────────────────────────────────────────

def write_code(state: CodeReviewState) -> dict:
    prompt = f"""Write clean Python code for the following task:
{state['task_description']}

Return ONLY the Python code, no explanations."""
    response = llm.invoke([HumanMessage(content=prompt)])
    print(f"[write_code] Generated {len(response.content)} chars of code")
    return {"code": response.content, "iteration": 0}

# ── NODE 2: REVIEW CODE ───────────────────────────────────────────────────────

def review_code(state: CodeReviewState) -> dict:
    prompt = f"""Review the following Python code for correctness, style, and edge cases.

CODE:
{state['code']}

Respond with:
SCORE: 
FEEDBACK: """
    response = llm.invoke([HumanMessage(content=prompt)])
    content = response.content

    # Parse score from response
    score = 5  # default
    for line in content.split("\n"):
        if line.startswith("SCORE:"):
            try:
                score = int(line.split(":")[1].strip())
            except ValueError:
                pass

    print(f"[review_code] Iteration {state['iteration']}, Score: {score}/10")
    return {
        "review_feedback": [content],
        "review_score": score,
        "iteration": state["iteration"] + 1,
    }

# ── NODE 3: FIX CODE ──────────────────────────────────────────────────────────

def fix_code(state: CodeReviewState) -> dict:
    last_feedback = state["review_feedback"][-1]
    prompt = f"""Improve the following Python code based on the reviewer's feedback.

ORIGINAL CODE:
{state['code']}

REVIEWER FEEDBACK:
{last_feedback}

Return ONLY the improved Python code, no explanations."""
    response = llm.invoke([HumanMessage(content=prompt)])
    print(f"[fix_code] Fixed code on iteration {state['iteration']}")
    return {"code": response.content}

# ── ROUTING ───────────────────────────────────────────────────────────────────

def route_after_review(state: CodeReviewState) -> str:
    MAX_ITERATIONS = 3
    score = state.get("review_score", 0)
    iteration = state.get("iteration", 0)

    if score >= 7 or iteration >= MAX_ITERATIONS:
        print(f"[router] Accepting code (score={score}, iteration={iteration})")
        return "accept"
    print(f"[router] Requesting fix (score={score}, iteration={iteration})")
    return "fix"

def finalize(state: CodeReviewState) -> dict:
    return {"final_code": state["code"]}

# ── GRAPH ─────────────────────────────────────────────────────────────────────

workflow = StateGraph(CodeReviewState)

workflow.add_node("write_code", write_code)
workflow.add_node("review_code", review_code)
workflow.add_node("fix_code", fix_code)
workflow.add_node("finalize", finalize)

workflow.add_edge(START, "write_code")
workflow.add_edge("write_code", "review_code")
workflow.add_conditional_edges(
    "review_code",
    route_after_review,
    {"fix": "fix_code", "accept": "finalize"}
)
workflow.add_edge("fix_code", "review_code")   # loop: fix → review again
workflow.add_edge("finalize", END)

app = workflow.compile(checkpointer=MemorySaver())

# ── RUN ───────────────────────────────────────────────────────────────────────

if __name__ == "__main__":
    config = {"configurable": {"thread_id": "code-review-1"}}
    result = app.invoke(
        {
            "task_description": "Write a function that returns the nth Fibonacci number using memoization",
            "code": "",
            "review_feedback": [],
            "review_score": 0,
            "iteration": 0,
            "final_code": None,
        },
        config=config
    )
    print("\n=== FINAL CODE ===")
    print(result["final_code"])
    print(f"\nTotal review iterations: {result['iteration']}")
    print(f"Final score: {result['review_score']}/10")

8 Common Mistakes
Mistake 1: Not initializing all state fields when invoking
If your TypedDict has a field like iteration: int, you must pass {"iteration": 0} in the initial invoke call. Missing fields cause KeyError inside nodes that access them.

Fix: Always provide a complete initial state dict, or use Optional[int] = None with None-checks in your nodes.
Mistake 2: Conditional edge routing function returns wrong values
Your routing function must return exactly the string keys you define in the add_conditional_edges mapping. Returning "end" when the key is END (which equals "__end__") silently fails or raises a routing error.

Fix: Always import and use END from langgraph.graph as the return value and as the mapping key.
Mistake 3: Forgetting to call .compile()
You cannot invoke a StateGraph directly — you must call app = workflow.compile() first. Calling workflow.invoke() raises an AttributeError.
Mistake 4: Using replace semantics on a list field without Annotated
If you define messages: List[str] without Annotated[..., operator.add], every node that returns {"messages": [new_msg]} replaces the entire list instead of appending. You lose all history.
Warning — Infinite loops: If your conditional routing function never returns END (e.g., the score never reaches 7), the graph runs forever. Always include a max_iterations guard in routing functions.
9 Exam Prep — Key Concepts
Q: What is the core data structure in LangGraph?
A: StateGraph — a directed graph where nodes are functions and edges define control flow. State is a typed dictionary shared across all nodes.
Q: How does LangGraph support human-in-the-loop?
A: Via checkpointing + interrupts. Compile with interrupt_before=["node_name"] to pause before a node. The graph saves state to the checkpointer, and execution resumes when you call app.invoke(None, config) with the same thread_id.
Q: What does thread_id do?
A: It groups a series of invocations into a single session. Each thread has its own checkpointed state. Two users with different thread_ids have completely independent state histories.
Q: What is the difference between add_edge and add_conditional_edges?
A: add_edge(A, B) always routes from A to B. add_conditional_edges(A, router_fn, mapping) calls router_fn(state) and uses the returned string to look up the next node in the mapping dict.
Q: What checkpointer is used in production vs. testing?
A: MemorySaver (in-memory, lost on restart) for testing/development. PostgresSaver or SqliteSaver for production persistence.
