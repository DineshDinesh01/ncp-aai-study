Production Agent Error Handling

1Why Error Handling Is Critical

A production agent calls external tools (APIs, databases, code execution environments) and LLM endpoints, all of which can fail. Without explicit error handling:

  • One failed tool call crashes the entire agent run
  • Context windows silently overflow, causing truncated reasoning
  • LLM rate limits cause hard failures instead of polite retries
  • In multi-agent systems, one subagent failure propagates to the orchestrator
  • Users see cryptic errors instead of helpful fallback responses
🎯 Exam Angle

The exam tests production reasoning: given a scenario where a tool fails or context overflows, which pattern should the agent use? Know the difference between retry, fallback, degradation, and escalation.

3Retry Logic with Exponential Backoff

For transient errors (rate limits, timeouts), the standard pattern is exponential backoff with jitter: wait 2^n seconds between retries, plus random jitter to avoid thundering-herd problems when many agents retry simultaneously.

Python — Retry Decorator for Tool Calls
import asyncio
import random
import logging
from functools import wraps
from openai import RateLimitError, APITimeoutError

logger = logging.getLogger(__name__)

def with_retry(max_retries=3, base_delay=1.0, max_delay=60.0):
    """Exponential backoff with jitter for transient errors."""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(max_retries + 1):
                try:
                    return await func(*args, **kwargs)

                except (RateLimitError, APITimeoutError) as e:
                    last_exception = e
                    if attempt == max_retries:
                        logger.error(f"All {max_retries} retries exhausted: {e}")
                        raise

                    # Exponential backoff + jitter
                    delay = min(base_delay * (2 ** attempt), max_delay)
                    jitter = random.uniform(0, delay * 0.1)
                    wait_time = delay + jitter

                    logger.warning(
                        f"Attempt {attempt+1}/{max_retries} failed ({type(e).__name__}). "
                        f"Retrying in {wait_time:.1f}s..."
                    )
                    await asyncio.sleep(wait_time)

                except Exception as e:
                    # Non-retriable errors: fail immediately
                    logger.error(f"Non-retriable error: {e}")
                    raise
        return wrapper
    return decorator

# Usage: wrap any tool call or LLM call
@with_retry(max_retries=3, base_delay=2.0)
async def call_web_search(query: str) -> str:
    response = await search_client.search(query)
    return response.results

@with_retry(max_retries=5, base_delay=1.0)  # More retries for LLM calls
async def call_llm(messages: list) -> str:
    response = await llm_client.chat.completions.create(
        model="meta/llama-3.1-70b-instruct",
        messages=messages,
    )
    return response.choices[0].message.content

4Timeout Handling

Agents must enforce timeouts at two levels: per-tool-call and per-agent-run.

Python — Per-Tool and Per-Run Timeouts
import asyncio

# Per-tool-call timeout
async def safe_tool_call(tool_fn, *args, timeout_seconds=10, **kwargs):
    """Wraps any tool call with a timeout. Returns error string on timeout."""
    try:
        result = await asyncio.wait_for(
            tool_fn(*args, **kwargs),
            timeout=timeout_seconds
        )
        return result
    except asyncio.TimeoutError:
        return f"ERROR: Tool call timed out after {timeout_seconds}s. Use cached result or skip."
    except Exception as e:
        return f"ERROR: Tool call failed — {type(e).__name__}: {str(e)[:200]}"

# LangGraph node with timeout
async def research_node(state: AgentState) -> AgentState:
    query = state["messages"][-1].content

    # Tool calls with individual timeouts
    search_result = await safe_tool_call(
        web_search, query, timeout_seconds=15
    )
    code_result = await safe_tool_call(
        code_executor, state.get("code"), timeout_seconds=30
    )

    return {"search_result": search_result, "code_result": code_result}

# Per-agent-run timeout (LangGraph level)
async def run_agent_with_timeout(graph, inputs, timeout_seconds=120):
    """Run the full agent graph with a global timeout."""
    try:
        result = await asyncio.wait_for(
            graph.ainvoke(inputs),
            timeout=timeout_seconds
        )
        return result
    except asyncio.TimeoutError:
        # Return partial state + human escalation flag
        return {
            "status": "timeout",
            "message": "Agent exceeded time limit. Escalating to human.",
            "partial_result": "..."
        }
5Context Overflow Strategies

Context overflow is the most common silent failure mode in production agents. When tool outputs are large or conversations grow long, the token count approaches the model's limit.

StrategyHow It WorksWhen to Use
Sliding windowKeep only the last N messages; drop oldestSimple chat agents; when history order matters less
SummarizationLLM compresses old turns into a summary; replace with summaryLong-running agents where historical context matters
Tool output truncationCap tool outputs at N tokens; add "...truncated" markerWeb search results, large API responses
RAG over historyEmbed past messages; retrieve relevant history per turnVery long sessions (100+ turns)
Python — Token-Aware Message Management
import tiktoken

def count_tokens(messages: list, model="gpt-4") -> int:
    enc = tiktoken.encoding_for_model(model)
    total = 0
    for msg in messages:
        total += len(enc.encode(msg.get("content", ""))) + 4  # per-message overhead
    return total

def manage_context(messages: list, max_tokens: int = 3000) -> list:
    """Trim messages to fit within token budget using sliding window + summarization."""
    if count_tokens(messages) <= max_tokens:
        return messages

    # Always keep system message + last 2 messages
    system_msgs = [m for m in messages if m["role"] == "system"]
    recent_msgs = messages[-2:]

    # Summarize the middle
    middle_msgs = messages[len(system_msgs):-2]
    if middle_msgs:
        summary_text = summarize_messages(middle_msgs)  # Call LLM to summarize
        summary_msg = {"role": "assistant", "content": f"[Summary of earlier context]: {summary_text}"}
        return system_msgs + [summary_msg] + recent_msgs

    return system_msgs + recent_msgs

def truncate_tool_output(output: str, max_chars: int = 2000) -> str:
    """Prevent large tool outputs from blowing context."""
    if len(output) <= max_chars:
        return output
    return output[:max_chars] + f"\n... [truncated {len(output)-max_chars} chars] ..."
6Graceful Degradation

Graceful degradation means returning a partial but useful response instead of a hard failure when something goes wrong. The agent acknowledges the limitation and provides what it can.

Python — Fallback Tool + Degraded Response
async def search_with_fallback(query: str) -> dict:
    """Try primary tool, fall back to cached/simplified alternative."""
    # Attempt 1: Real-time web search
    try:
        result = await web_search(query, timeout=10)
        return {"source": "live_search", "result": result, "degraded": False}
    except Exception as e:
        logger.warning(f"Live search failed: {e}")

    # Fallback 1: Cached results from vector store
    try:
        cached = await vector_store.similarity_search(query, k=3)
        if cached:
            return {
                "source": "cache",
                "result": "\n".join([d.page_content for d in cached]),
                "degraded": True,
                "degradation_reason": "Live search unavailable; using cached knowledge"
            }
    except Exception as e:
        logger.warning(f"Cache lookup failed: {e}")

    # Fallback 2: LLM knowledge only (no retrieval)
    return {
        "source": "llm_only",
        "result": None,
        "degraded": True,
        "degradation_reason": "All retrieval unavailable; responding from training knowledge only"
    }

# In the agent node: surface degradation to user
async def agent_node(state):
    search = await search_with_fallback(state["query"])

    if search["degraded"]:
        # Inject degradation notice into prompt
        system_addon = f"\nNOTE: {search['degradation_reason']}. Be transparent about this."
        state["system_note"] = system_addon

    return state
7Circuit Breakers in Multi-Agent Systems

In multi-agent systems, a failing subagent should not cascade into the orchestrator. A circuit breaker tracks failure rates and temporarily stops calling a failing service, preventing cascading failures across the system.

Python — Simple Circuit Breaker
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "closed"       # Normal operation — calls go through
    OPEN = "open"           # Too many failures — block all calls
    HALF_OPEN = "half_open" # Testing if service recovered

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=60):
        self.failure_threshold = failure_threshold  # Failures before opening
        self.recovery_timeout = recovery_timeout    # Seconds before trying again
        self.failure_count = 0
        self.last_failure_time = None
        self.state = CircuitState.CLOSED

    def call(self, func, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            # Check if recovery timeout has passed
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
            else:
                raise Exception(f"Circuit OPEN: subagent unavailable. Try again in {self.recovery_timeout}s")

        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_success(self):
        self.failure_count = 0
        self.state = CircuitState.CLOSED

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN

# Usage in multi-agent orchestrator
research_breaker = CircuitBreaker(failure_threshold=3, recovery_timeout=30)
code_breaker = CircuitBreaker(failure_threshold=5, recovery_timeout=60)

def orchestrator_node(state):
    try:
        research = research_breaker.call(research_agent, state["query"])
    except Exception:
        research = "Research subagent unavailable — using cached knowledge."

    try:
        code = code_breaker.call(code_agent, state["task"])
    except Exception:
        code = "Code execution unavailable — providing pseudocode only."

    return {"research": research, "code": code}

9Exam Prep
🎯 Exam Key Points — Production Error Handling
  • Transient errors (429, 503, timeout) → exponential backoff with jitter; NOT immediate fail
  • Context overflow → summarize old turns OR sliding window; never just truncate blindly
  • Tool failure → fallback to cached result or alternative tool; surface degradation to user
  • Circuit breaker → CLOSED (normal) → OPEN (blocking) → HALF_OPEN (testing) states
  • Exponential backoff formula: delay = min(base * 2^attempt, max_delay) + random jitter
  • Jitter purpose: prevent thundering herd — many agents retrying at exact same second
  • Graceful degradation: return partial result + acknowledgment rather than error exception
  • Max iterations: agents need hard iteration limits (max_iter) to prevent infinite tool loops
  • Timeout hierarchy: per-tool-call timeout < per-agent-run timeout < system-level timeout
