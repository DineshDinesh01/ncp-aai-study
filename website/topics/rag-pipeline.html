RAG Pipeline

Retrieval-Augmented Generation gives your LLM access to external knowledge — documents, databases, APIs — without retraining. It's the backbone of most production AI agents.

1Definition

RAG (Retrieval-Augmented Generation) is a technique where an LLM's response is grounded in documents retrieved at query time from an external knowledge source. Instead of relying solely on what the model learned during training, RAG fetches relevant information and injects it into the prompt.

The result: your LLM can answer questions about documents it never saw during training, always uses up-to-date information, and can cite its sources.

2Why It Exists

Problem 1 — Knowledge cutoff: LLMs are trained on data up to a certain date. They cannot answer questions about events after that cutoff.

Problem 2 — Hallucination: When an LLM doesn't know something, it often invents a plausible-sounding but incorrect answer.

Problem 3 — Private data: LLMs cannot know about your company's internal documents, proprietary databases, or confidential information.

RAG solves all three. By retrieving real documents and placing them in the prompt, the LLM is grounded in factual, current, private knowledge.

3How It Works (Step-by-Step)

Offline Phase (Index Building):

Step 1 — Load documents: PDFs, Word docs, web pages, Confluence pages → raw text

Step 2 — Chunk: Split text into overlapping chunks (e.g., 512 tokens with 50-token overlap). Chunk size matters — too small loses context, too large dilutes relevance.

Step 3 — Embed: Each chunk is converted to a dense vector (embedding) using an embedding model (e.g., text-embedding-3-small, NVIDIA NeMo Retriever). These vectors encode semantic meaning.

Step 4 — Store: Vectors + metadata stored in a vector database (ChromaDB, Milvus, FAISS, pgvector).

Online Phase (Query Time):

Step 5 — Embed query: The user's question is embedded using the same embedding model.

Step 6 — Retrieve: ANN (Approximate Nearest Neighbor) search finds top-k most similar chunks from the vector store.

Step 7 — Augment: Retrieved chunks are injected into the prompt: "Given the following context: [chunks]... Answer: [user question]"

Step 8 — Generate: LLM generates an answer grounded in the retrieved context.

5Implementation

pythonrag_pipeline.py
# pip install llama-index-core llama-index-vector-stores-chroma chromadb
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.core import StorageContext
import chromadb

# Step 1: Load documents (put your PDFs/text files in ./docs/)
documents = SimpleDirectoryReader("./docs").load_data()
print(f"Loaded {len(documents)} documents")

# Step 2: Set up ChromaDB vector store (free, local)
chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection = chroma_client.get_or_create_collection("study_notes")
vector_store = ChromaVectorStore(chroma_collection=collection)
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# Step 3: Build the index (chunks + embeds + stores automatically)
index = VectorStoreIndex.from_documents(
    documents,
    storage_context=storage_context,
    show_progress=True
)

# Step 4: Query the index
query_engine = index.as_query_engine(similarity_top_k=3)
response = query_engine.query("What are the main topics covered?")

print(f"\nAnswer: {response}")
print(f"\nSource nodes used:")
for node in response.source_nodes:
    print(f"  Score: {node.score:.3f} | {node.text[:100]}...")
💡 Key Parameter: similarity_top_k

similarity_top_k=3 means retrieve the 3 most similar chunks. Increase for more context, decrease for precision. Typical values: 3–5 for QA, 8–10 for summarization.

7Hands-On Lab

Objective: Build a Q&A bot over a PDF document. Measure how chunk size affects retrieval quality.

pythonlab_rag.py
"""
LAB: RAG Q&A Bot — compare chunk sizes
Run on: Google Colab (free)
Install: pip install llama-index-core llama-index-vector-stores-chroma chromadb
"""
from llama_index.core import VectorStoreIndex, Document
from llama_index.core.node_parser import SentenceSplitter
import chromadb
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.core import StorageContext

# Sample document (replace with your own PDF)
text = """
NVIDIA Triton Inference Server is an open-source inference serving software.
It supports multiple frameworks including TensorFlow, PyTorch, and ONNX.
Triton uses dynamic batching to maximize GPU utilization.
The HTTP endpoint runs on port 8000, gRPC on port 8001.
NeMo Guardrails is an open-source toolkit for adding programmable guardrails to LLMs.
It uses Colang, a domain-specific language for dialogue flow definition.
The five rail types are: input, dialog, retrieval, execution, and output rails.
"""

questions = [
    "What port does Triton use for HTTP?",
    "What is Colang used for?",
    "How does Triton maximize GPU usage?"
]

# Test different chunk sizes
for chunk_size in [128, 256, 512]:
    print(f"\n{'='*50}")
    print(f"Chunk size: {chunk_size}")
    
    chroma = chromadb.Client()
    col = chroma.get_or_create_collection(f"test_{chunk_size}")
    vs = ChromaVectorStore(chroma_collection=col)
    sc = StorageContext.from_defaults(vector_store=vs)
    
    parser = SentenceSplitter(chunk_size=chunk_size, chunk_overlap=20)
    nodes = parser.get_nodes_from_documents([Document(text=text)])
    print(f"  Chunks created: {len(nodes)}")
    
    index = VectorStoreIndex(nodes, storage_context=sc)
    qe = index.as_query_engine(similarity_top_k=2)
    
    for q in questions:
        r = qe.query(q)
        print(f"  Q: {q}")
        print(f"  A: {str(r)[:100]}")
        
# VERIFY: Smaller chunks = more precise retrieval for specific facts
# VERIFY: Larger chunks = better for context-heavy questions

8Common Mistakes

❌ Mistake 1 — Wrong Chunk Size

Chunks too small (50 tokens): lose sentence context, retrieval misses related info. Chunks too large (2048 tokens): dilute relevance, slow retrieval. Fix: Start with 512 tokens, 10% overlap, then tune based on your retrieval quality metrics.

❌ Mistake 2 — Embedding Model Mismatch

Using a different embedding model at query time vs. indexing time produces incompatible vectors — similarity search fails silently and returns garbage results. Fix: Always use the same embedding model for both indexing and querying. Pin the model version.

❌ Mistake 3 — Not Filtering Irrelevant Chunks

Injecting low-similarity chunks into the prompt confuses the LLM and introduces wrong information. Fix: Add a similarity threshold (e.g., score > 0.7) and filter chunks below it before injecting.
