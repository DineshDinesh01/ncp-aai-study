1. InfiniBand Overview

InfiniBand (IB) is a high-bandwidth, low-latency network interconnect used to connect DGX nodes in AI clusters. Unlike Ethernet, IB uses RDMA (Remote Direct Memory Access), allowing GPUs to transfer data between nodes without CPU involvement — critical for distributed training all-reduce operations.

Why InfiniBand for AI Clusters?
NCCL (NVIDIA Collective Communications Library) uses IB RDMA for all-reduce and all-gather operations in model-parallel training. IB's microsecond latency vs Ethernet's millisecond latency directly impacts training throughput at scale.

HDR Split Ports
HDR switches often use 2× HDR100 (100 Gb/s) ports instead of one HDR (200 Gb/s) port. When connecting to older switches, you may see Active Width: 1X at 100 Gb/s. Always check ibstat for actual active speed.

4. ibstat — Port Status

# Show status of all IB ports
ibstat

# Example output for a healthy port:
# CA 'mlx5_0'
#   Port 1:
#     State: Active          ← MUST be Active for connectivity
#     Physical state: LinkUp
#     Rate: 200              ← HDR; should be 400 for NDR
#     Base lid: 12
#     LMC: 0
#     SM lid: 1
#     Capability mask: 0x...

# Check specific HCA
ibstat mlx5_0
ibstat mlx5_0 1            # Port 1 of mlx5_0

# List all IB devices
ibstat | grep "CA '"
State vs Physical State
  • State: Active = data can flow
  • State: Down = cable disconnected or SM not running
  • Physical state: Polling = trying to connect (no active neighbor)
  • Physical state: LinkUp but State: Init = SM not finished LID assignment

5. iblinkinfo — Fabric-Wide Link View

# Show all links in the IB fabric
iblinkinfo

# Filter to show only active links
iblinkinfo -a

# Show links for specific switch by GUID
iblinkinfo -S 0x0002c9000012345

# Compact output with speed/width info
iblinkinfo --line

# Check for link speed mismatches (degraded links)
iblinkinfo | grep -v "Active"         # Find non-active links
iblinkinfo | grep "Downgraded"        # Links running at lower than max speed

Use iblinkinfo to identify degraded links (running at less than rated speed) which cause asymmetric NCCL performance in distributed training.

6. ibdiagnet — Fabric Diagnostics

# Full fabric diagnostic scan
ibdiagnet

# Specific checks:
ibdiagnet --pc              # Check performance counters for errors
ibdiagnet --pm              # Check port counters
ibdiagnet -lw               # Check link width (flag degraded 1x links)
ibdiagnet -ls               # Check link speed
ibdiagnet --sm              # Check subnet manager configuration

# Generate full report
ibdiagnet -o /tmp/ibdiag_report

# Check for bad cables or links
ibdiagnet --pc 2>/dev/null | grep -i "error\|fail"

7. perfquery — Performance Counters

# Query performance counters for all ports on a node
perfquery

# Query specific port (lid=12, port=1)
perfquery 12 1

# Show extended counters (64-bit)
perfquery -x 12 1

# Monitor a specific error counter
perfquery | grep -i "PortRcvErrors\|PortXmitDiscards\|LinkDowned"

# Key counters to watch for link problems:
# PortRcvErrors     — receive-side errors (bad cable)
# PortXmitDiscards  — transmit drops (congestion)
# LinkDowned        — how many times the link went down
# SymbolErrors      — electrical signal errors

# Reset counters after clearing issues
perfquery -R 12 1

8. NCCL Testing

NCCL tests validate all-reduce bandwidth across multiple DGX nodes — the real-world test of IB fabric performance:

# Install nccl-tests
git clone https://github.com/NVIDIA/nccl-tests.git
cd nccl-tests && make MPI=1 MPI_HOME=/usr/local/mpi CUDA_HOME=/usr/local/cuda

# Single-node all-reduce test (8 GPUs)
./build/all_reduce_perf -b 8 -e 8G -f 2 -g 8

# Multi-node all-reduce test (2 nodes × 8 GPUs = 16 GPUs total)
mpirun -np 16 -N 8 \
  --hostfile hostfile \
  -x NCCL_DEBUG=INFO \
  -x NCCL_IB_HCA=mlx5 \
  ./build/all_reduce_perf -b 8 -e 8G -f 2 -g 8

# Key NCCL environment variables for IB:
# NCCL_IB_HCA=mlx5       — Use all mlx5 InfiniBand HCAs
# NCCL_IB_GID_INDEX=3    — RDMA GID index (use 3 for RoCE v2)
# NCCL_SOCKET_IFNAME=ib0 — Fallback socket interface
# NCCL_DEBUG=INFO        — Verbose output for troubleshooting
Expected Bandwidth
For DGX H100 with NDR (400 Gb/s × 8 ports): all-reduce should achieve ~300-380 Gb/s effective bandwidth per GPU in a well-configured fabric. Significantly lower numbers indicate IB fabric issues.

9. Exam Prep & Quick Reference

Must-Know for Exam
  • DGX A100 = HDR (200 Gb/s) × 8 ports | DGX H100 = NDR (400 Gb/s) × 8 ports
  • Subnet Manager must run exactly once per IB subnet (usually on a switch)
  • ibstat checks port State (must be Active) and speed
  • iblinkinfo shows fabric-wide link status; use to find degraded/down links
  • ibdiagnet --pc checks performance counter errors across whole fabric
  • perfquery reads per-port error counters: PortRcvErrors, LinkDowned
  • NCCL tests verify end-to-end all-reduce bandwidth between nodes
  • NCCL_IB_HCA=mlx5 tells NCCL to use InfiniBand HCAs (not Ethernet)
