Transformer Architecture

The transformer is the foundational architecture behind every modern large language model — from BERT and GPT to Llama and Mixtral. Understanding how attention, positional encoding, and feed-forward layers work together is essential for the NCP-AAI exam and for building production AI systems on NVIDIA hardware.

1 What Is a Transformer?

A transformer is a neural network architecture introduced by Vaswani et al. in 2017 in the paper "Attention Is All You Need." It completely discards the recurrent and convolutional layers that dominated NLP before it, replacing them with a mechanism called self-attention that can relate any token in a sequence to any other token in a single operation — regardless of distance.

The original transformer has two halves:

  • Encoder — reads the full input sequence and builds a rich contextual representation of it. Used in models like BERT that need to understand text (classification, Q&A, named-entity recognition).
  • Decoder — auto-regressively generates output tokens one at a time, attending both to its own previous outputs and to the encoder's representation. Used in translation models.
  • Decoder-only — a simplified design (no encoder cross-attention) that is the basis for all GPT-family models, Llama, Mistral, and most modern LLMs used in agentic systems.

Each half is built from a stack of identical layers. GPT-2 Small has 12 layers; GPT-3 has 96. Each layer contains two main sub-modules:

👁️
Multi-Head Self-Attention
Computes how much each token should "attend to" every other token. Multiple attention heads run in parallel, each learning different relationship patterns (syntax, coreference, semantics).
⚡
Position-wise Feed-Forward Network
A two-layer MLP applied identically to each token position. This is where most of the model's factual knowledge is believed to be stored. Typically 4× the model dimension.

Both sub-modules are followed by Add & LayerNorm — a residual connection that adds the sub-module's input back to its output, then normalizes. This is critical for training deep stacks of layers without gradient vanishing.

2 Why Transformers Replaced RNNs

Before transformers, the dominant NLP architecture was the Recurrent Neural Network (RNN) — and its improved variants LSTM (Long Short-Term Memory) and GRU (Gated Recurrent Unit). RNNs process tokens sequentially: the hidden state from step t−1 is fed into step t.

This sequential dependency creates three severe problems:

Problem RNN / LSTM Transformer
Parallelism Must process tokens in order — cannot parallelize training across a sequence All token positions computed simultaneously — trains on entire sequence in one pass
Long-range dependencies Hidden state "forgets" information from hundreds of tokens ago despite LSTM gates Any token can attend directly to any other token — distance is irrelevant
Gradient flow Vanishing / exploding gradients over long sequences even with gating Residual connections provide stable gradient highways through all layers
GPU utilization Low — sequential ops can't saturate thousands of CUDA cores High — attention is a batch matrix multiply, perfect for tensor cores

The transformer's ability to parallelize made it possible to train on vastly larger datasets and scale to billions of parameters. The GPU utilization advantage directly explains why NVIDIA hardware (A100, H100, H200) has become the de-facto compute platform for LLM training and inference.

Key Insight

Transformers trade compute (attention is O(n²) in sequence length) for parallelism. For typical context lengths (up to ~128K tokens), modern hardware handles this trade-off well. Techniques like Flash Attention and sparse attention reduce the quadratic cost further.

3 Step-by-Step Forward Pass

Let's trace a sentence through a decoder-only transformer layer by layer. Suppose the input is "The cat sat".

Step 1 — Tokenization & Input Embedding

The input text is tokenized into sub-word tokens using a vocabulary (e.g., BPE). Each token ID is looked up in an embedding matrix of shape [vocab_size, d_model], producing a dense vector of dimension d_model (e.g., 768 for GPT-2 Small, 12288 for GPT-4). This converts discrete symbols into continuous floating-point space where the model can do algebra.

Step 2 — Positional Encoding

Self-attention is position-agnostic by default — it treats the input as a set, not a sequence. To inject order, a positional encoding vector is added to each token embedding. Two main flavors:

  • Sinusoidal (original paper) — fixed sine/cosine functions at different frequencies for each position. No learned parameters; generalizes to lengths not seen during training.
  • Learned absolute (BERT, GPT-2) — a trainable embedding per position up to a maximum context length.
  • Rotary Position Embedding — RoPE (Llama, Mistral) — encodes position by rotating query and key vectors. Enables better length extrapolation and is now the standard in most new architectures.

Step 3 — Multi-Head Self-Attention

This is the heart of the transformer. For each attention head, three linear projections are applied to the input:

  • Q (Query) — "What am I looking for?"
  • K (Key) — "What do I have to offer?"
  • V (Value) — "What information do I actually contain?"

Attention scores are computed as dot products of queries and keys, scaled by √d_k to prevent vanishing gradients in the softmax, then converted to probabilities:

MATH Scaled Dot-Product Attention 
Attention(Q, K, V) = softmax( Q·Kᵀ / √d_k ) · V

Where:
  Q  = input × W_Q    (shape: [seq_len, d_k])
  K  = input × W_K    (shape: [seq_len, d_k])
  V  = input × W_V    (shape: [seq_len, d_v])
  √d_k               (scaling factor to prevent tiny gradients)

Multi-head attention runs h independent attention heads in parallel, each with its own W_Q, W_K, W_V projections, then concatenates and linearly projects the results. Different heads tend to specialize — one might track subject-verb agreement, another coreference, another syntax structure.

In decoder-only models, a causal mask is applied before softmax: position i can only attend to positions ≤ i. This enforces autoregressive generation (no "cheating" by looking at future tokens).

Step 4 — Add & LayerNorm

The attention output is added back to the original input (residual connection), then normalized: LayerNorm(x + Attention(x)). The residual connection is not optional — without it, very deep networks cannot be trained. LayerNorm stabilizes activations across the feature dimension.

Step 5 — Position-wise Feed-Forward Network

Two linear layers with a non-linearity between them, applied independently to every token position: FFN(x) = max(0, xW₁ + b₁)W₂ + b₂. Modern models use SwiGLU or GELU activation instead of ReLU. The intermediate dimension is typically 4× d_model (e.g., 3072 for d_model=768). This is where most of the model's "knowledge" is stored.

Step 6 — Second Add & LayerNorm

Same residual + norm pattern: LayerNorm(x + FFN(x)). This completes one transformer block. Stack N blocks and you have a full transformer.

Step 7 — Output & Language Model Head

The final layer's output for the last token is projected through an LM Head — a linear layer mapping from d_model to vocab_size. A softmax gives a probability distribution over the entire vocabulary. The token with the highest probability (greedy decoding) or a sampled token (temperature/nucleus sampling) is selected as the next token, appended to the context, and the process repeats.

The transformer's architecture directly impacts every performance consideration downstream:

  • Context window — determined by the maximum sequence length the positional encoding supports. Longer context = more tokens in the KV cache = more GPU memory required.
  • KV Cache — during inference, the Key and Value matrices computed for previous tokens are cached. This avoids recomputing them at every generation step. Memory usage = 2 × layers × heads × d_head × context_length × bytes_per_element.
  • Model parallelism — very large models (70B+ parameters) are split across multiple GPUs either by layer (pipeline parallelism) or by attention head / tensor dimension (tensor parallelism). NVIDIA's NeMo and Megatron-LM implement both strategies.
  • Quantization — reducing weight precision from FP32 → FP16 → INT8 → INT4 reduces memory and increases throughput. NVIDIA TensorRT-LLM handles this automatically.

5 Code: HuggingFace Inference & Attention Weights

The following complete example loads GPT-2, runs text generation, and then extracts attention weights from all layers and heads using the output_attentions=True flag. This is the same technique used in the BertViz visualization library.

PYTHON transformer_inference.py 
"""
Transformer Architecture — HuggingFace Inference + Attention Weight Inspection
Requires: pip install transformers torch bertviz
"""

import torch
from transformers import GPT2Tokenizer, GPT2Model, GPT2LMHeadModel
import matplotlib.pyplot as plt
import numpy as np

# ─────────────────────────────────────────────────────────────────────────────
# 1. Basic inference with GPT-2
# ─────────────────────────────────────────────────────────────────────────────
tokenizer = GPT2Tokenizer.from_pretrained("gpt2")
lm_model = GPT2LMHeadModel.from_pretrained("gpt2")
lm_model.eval()  # disable dropout for deterministic output

prompt = "The transformer architecture changed natural language processing because"
inputs = tokenizer(prompt, return_tensors="pt")

with torch.no_grad():
    # max_new_tokens limits generation; do_sample=False = greedy decoding
    output_ids = lm_model.generate(
        **inputs,
        max_new_tokens=50,
        do_sample=False,       # greedy — always picks highest probability token
        temperature=1.0,       # ignored when do_sample=False
        pad_token_id=tokenizer.eos_token_id
    )

generated_text = tokenizer.decode(output_ids[0], skip_special_tokens=True)
print("Generated text:")
print(generated_text)
print()

# ─────────────────────────────────────────────────────────────────────────────
# 2. Extract attention weights from all layers and heads
#    output_attentions=True returns a tuple: one tensor per layer
#    Each tensor shape: [batch_size, num_heads, seq_len, seq_len]
# ─────────────────────────────────────────────────────────────────────────────
base_model = GPT2Model.from_pretrained("gpt2")
base_model.eval()

text = "The cat sat on the mat near the window"
enc = tokenizer(text, return_tensors="pt")
token_labels = tokenizer.convert_ids_to_tokens(enc["input_ids"][0])

with torch.no_grad():
    outputs = base_model(
        **enc,
        output_attentions=True  # ← request all attention matrices
    )

# outputs.attentions is a tuple of length num_layers (12 for gpt2)
# Each element: tensor of shape [1, num_heads, seq_len, seq_len]
attentions = outputs.attentions
print(f"Number of layers:   {len(attentions)}")
print(f"Heads per layer:    {attentions[0].shape[1]}")
print(f"Sequence length:    {attentions[0].shape[2]}")

# ─────────────────────────────────────────────────────────────────────────────
# 3. Visualize attention from layer 0, head 0
#    Rows = query tokens, Columns = key tokens
#    Value = how much the row token attends to the column token
# ─────────────────────────────────────────────────────────────────────────────
layer_idx = 0
head_idx  = 0
attn_matrix = attentions[layer_idx][0, head_idx].numpy()  # [seq_len, seq_len]

fig, ax = plt.subplots(figsize=(8, 7))
im = ax.imshow(attn_matrix, cmap="Greens", vmin=0, vmax=attn_matrix.max())

ax.set_xticks(range(len(token_labels)))
ax.set_yticks(range(len(token_labels)))
ax.set_xticklabels(token_labels, rotation=45, ha="right", fontsize=10)
ax.set_yticklabels(token_labels, fontsize=10)
ax.set_title(f"GPT-2  Layer {layer_idx}  Head {head_idx}\n(row token attends to column token)")
plt.colorbar(im, ax=ax)
plt.tight_layout()
plt.savefig("attention_heatmap.png", dpi=150)
plt.show()
print("Saved attention_heatmap.png")

# ─────────────────────────────────────────────────────────────────────────────
# 4. Average attention across all heads for a given layer
#    Useful for summarizing "where does this layer focus?"
# ─────────────────────────────────────────────────────────────────────────────
avg_attn = attentions[layer_idx][0].mean(dim=0).numpy()  # average over heads
print(f"\nLayer {layer_idx} average attention shape: {avg_attn.shape}")

# Which token does 'cat' (index 1 with Ġ prefix) attend to most on average?
cat_idx = 1
top_k = np.argsort(avg_attn[cat_idx])[::-1][:3]
print(f"\n'cat' (index {cat_idx}) attends most to:")
for idx in top_k:
    print(f"  [{idx}] {token_labels[idx]:15s}  score={avg_attn[cat_idx][idx]:.4f}")

6 NVIDIA NeMo Toolkit Mapping

NVIDIA's NeMo Toolkit is the production framework for building, training, and fine-tuning transformer models at scale on NVIDIA hardware. It sits on top of PyTorch Lightning and provides ready-made transformer implementations optimized for multi-GPU and multi-node training.

NVIDIA NeMo Toolkit on GitHub

Feed-Forward Network FusedMLP with SwiGLU activation Fused CUDA kernel for speed; intermediate = 4× or 8/3× d_model

Full Model Config MegatronGPTModel YAML config Controls num_layers, hidden_size, num_attention_heads, seq_length

NeMo uses Megatron-LM as its core for large-scale training. Megatron-LM was developed at NVIDIA and introduced tensor parallelism — splitting individual transformer layers across multiple GPUs rather than assigning whole layers to each GPU. This allows training models that are too large to fit in a single GPU's memory even with pipeline parallelism alone.

NeMo + NVIDIA NIMs Connection

Models trained or fine-tuned with NeMo can be exported to NVIDIA NIMs (NVIDIA Inference Microservices) for deployment. NIMs wrap the model in a containerized, optimized inference server with a standard OpenAI-compatible API. This is the recommended production path for NVIDIA Agentic AI deployments.

8 Common Mistakes

Mistake 1 — Confusing Encoder-Only vs Decoder-Only

BERT is encoder-only: it sees the entire sequence bidirectionally and produces contextual embeddings. It cannot generate text autoregressively. GPT is decoder-only: it generates text left-to-right with causal masking. Using BERT for generation or GPT for classification (without a head) are both incorrect architectural choices. On the exam, know which tasks each variant is designed for.

Mistake 2 — Forgetting Positional Encoding Is Not Optional

Removing positional encoding means the model receives a set, not a sequence. The output would be identical regardless of word order — "the dog bit the man" and "the man bit the dog" would produce identical representations. Always include positional encoding; in modern models, RoPE is applied inside the attention computation, not added to embeddings.

Mistake 3 — Thinking Larger d_model Always Means Better Performance

Scaling is about the ratio of parameters to training data, not just model size. A large model under-trained on few tokens (not following the Chinchilla scaling laws) underperforms a smaller, well-trained model. The NCP-AAI exam tests understanding of training compute efficiency, not just parameter counts.

Mistake 4 — Misunderstanding the KV Cache

The KV cache stores Keys and Values (not Queries) from previous tokens. Queries are computed fresh for each new token because they represent "what the current token is looking for." Misidentifying what is cached is a common exam trap.

Mistake 5 — Treating Attention as Explainability

High attention weight between two tokens does not definitively mean one causally depends on the other. Research has shown attention can be manipulated without changing model outputs. Attention is a mechanism, not a transparent window into model reasoning.
