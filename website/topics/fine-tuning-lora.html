Fine-Tuning with LoRA

Low-Rank Adaptation (LoRA) and QLoRA — the parameter-efficient fine-tuning techniques behind NeMo Customizer. Understand the math, implement SFT, choose between LoRA/QLoRA/DPO, and configure NeMo Customizer for production agents.

1What Is LoRA?

Low-Rank Adaptation (LoRA) is a parameter-efficient fine-tuning technique that freezes the original model weights and injects small trainable rank-decomposition matrices into each target layer. Instead of updating all W ∈ ℝd×d (billions of parameters), LoRA trains only two small matrices A ∈ ℝd×r and B ∈ ℝr×d where r ≪ d.

During forward pass: W' = W + BA where W is frozen and only BA is trained. At inference, BA is merged back into W — zero added latency.

2Why LoRA Exists

Full fine-tuning a 70B model requires ~560 GB of GPU memory (FP32 weights + gradients + optimizer states). LoRA at rank r=16 trains only 0.1–1% of parameters, reducing memory from 560 GB to ~20 GB — enabling fine-tuning on a single H100.

3Internal Working

Rank r is the key hyperparameter. Higher r = more capacity = more parameters to train. Common values: r=4 (minimal), r=8 (baseline), r=16 (standard), r=64 (high-capacity).

Alpha (α) scales the LoRA output: effective update = (α/r) × BA. Setting α=2r (e.g., r=16, α=32) is a common initialization.

Target modules — LoRA is applied to specific weight matrices. For attention layers: q_proj, k_proj, v_proj, o_proj. Adding gate/up/down (MLP layers) improves task performance.

QLoRA adds 4-bit NF4 quantization of the frozen base model weights, further halving memory. Gradients are computed in BF16 for stability.

4SFT vs DPO — Choosing the Right Method
MethodTraining SignalData FormatBest For
SFT (Supervised Fine-Tuning)Next-token prediction on demonstrations{prompt, completion} pairsTeaching new tasks, format following

NeMo Customizer supports both SFT and DPO natively. For most agentic use cases, SFT with LoRA is the recommended starting point.

5Implementation
Python — LoRA with HuggingFace PEFT

from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from peft import LoraConfig, get_peft_model, TaskType
from trl import SFTTrainer
import torch

model_id = "meta-llama/Meta-Llama-3-8B-Instruct"
model = AutoModelForCausalLM.from_pretrained(
    model_id, torch_dtype=torch.bfloat16, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(model_id)

# LoRA configuration
lora_config = LoraConfig(
    r=16,                          # rank
    lora_alpha=32,                 # scaling factor (alpha = 2*r)
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type=TaskType.CAUSAL_LM
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 41,943,040 || all params: 8,072,204,288 || 0.5197%

# Training

8Common Mistakes
❌ Choosing Too Low a Rank

r=1 or r=2 often underfits. Start with r=8 or r=16 for agent task adaptation.

❌ Forgetting to Set alpha

Default alpha=1 in some libraries creates tiny updates. Set alpha=2r (e.g., r=16 → alpha=32).

❌ Not Merging Adapters for Deployment

Un-merged LoRA adds compute overhead on every forward pass. Call model.merge_and_unload() before exporting to TensorRT-LLM/NIMs.
