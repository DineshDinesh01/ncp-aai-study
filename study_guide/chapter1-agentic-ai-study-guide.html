🟣 LangGraph Activity
🛠️ Build a Chainlit chat UI with HITL interrupt

Create a simple agent that can search the web, but pauses and asks the user to confirm before executing the search.

  1. Install: pip install chainlit langgraph langchain-openai
  2. Create a StateGraph with a tool-calling node and a human approval node
  3. Use interrupt_before=["tool_node"] when compiling the graph
  4. In Chainlit, call cl.AskActionMessage to show approve/reject buttons
  5. Resume the graph with the user's choice using graph.invoke(None, config)
# app.py (Chainlit entry point)
import chainlit as cl
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.prebuilt import ToolNode, tools_condition

def my_tool(query: str) -> str:
    """Search the web for a query."""
    return f"Results for: {query}"

# Build graph with interrupt before tools
builder = StateGraph(MessagesState)
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode([my_tool]))
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", tools_condition)
builder.add_edge("tools", "agent")

graph = builder.compile(interrupt_before=["tools"])

@cl.on_message
async def main(msg: cl.Message):
    # invoke graph, it will pause before tools
    state = graph.invoke({"messages": [msg.content]})
    # ask user to approve
    res = await cl.AskActionMessage(
        content="Agent wants to run a tool. Approve?",
        actions=[cl.Action(name="approve", label="✅ Approve"),
                 cl.Action(name="reject", label="❌ Reject")]
    ).send()
    if res.get("name") == "approve":
        final = graph.invoke(None, config)  # resume
        await cl.Message(content=final["messages"][-1].content).send()

Think → Act → Observe loop · Tool binding · Agent executor
Exam Topic create_react_agent AgentIQ react_agent
▼
📖 What to Study
  • ReAct = Reasoning + Acting: the agent alternates between thinking in natural language (Thought) and calling tools (Action), then observing the result (Observation)
  • The loop continues until the agent decides it has enough information to give a Final Answer
  • Tool binding: how you attach Python functions to an LLM as callable tools
  • @tool decorator in LangChain — docstrings become the tool description the LLM reads
  • Difference between a simple LLM chain and a ReAct agent loop (the loop can run multiple times)
  • Stopping conditions: max iterations, forced output format, explicit stop signal

🟣 LangGraph Activity
🛠️ Build a 2-tool ReAct agent and trace the loop

Build a ReAct agent with a calculator tool and a mock search tool. Print each Thought/Action/Observation step.

  1. Define two tools using the @tool decorator — one calculator, one web stub
  2. Use create_react_agent(llm, tools) from langgraph.prebuilt
  3. Invoke with a multi-step question (e.g. "What is 25% of the population of India?")
  4. Stream with graph.stream() and print each message type
  5. Count how many loop iterations the agent took — adjust max_iterations
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def calculator(expression: str) -> str:
    """Evaluate a math expression. Input: Python math string."""
    return str(eval(expression))

@tool
def search(query: str) -> str:
    """Search for a fact. Returns a stub answer."""
    return "India population is approximately 1.44 billion"

llm = ChatOpenAI(model="gpt-4o-mini")
agent = create_react_agent(llm, [calculator, search])

# Stream and print every step
for chunk in agent.stream(
    {"messages": [("user", "What is 25% of India's population?")]}
):
    for node, state in chunk.items():
        print(f"\n[{node}]", state["messages"][-1].content)

Configure Agent-to-Agent Communication Protocols
Supervisor patterns · Message passing · MCP protocol
Exam Topic LangGraph Multi-Agent AgentIQ MCP
▼
📖 What to Study
  • Supervisor pattern: one orchestrator agent routes tasks to specialized worker agents
  • Message passing vs. shared state: agents communicate by reading/writing to a shared state dict
  • Handoff: one agent passes control to another by returning a specific message or tool call
  • Model Context Protocol (MCP): open standard for tool servers — agents connect to MCP servers to use external tools
  • AgentIQ treats agent communication over HTTP/gRPC (not a custom protocol) — aligns with MCP
  • Trust levels between agents: how much should a sub-agent's output be trusted without verification?

1.4
Manage Short-Term and Long-Term Memory
In-context memory · Persistent stores · Semantic retrieval
Exam Topic MemorySaver · Checkpointer MemMachine Integration
▼
📖 What to Study
  • Short-term memory = the current conversation context window. Lives only for one session.
  • Long-term memory = persisted outside the LLM. Survives across sessions. Stored in DB, vector store, or key-value store.
  • LangGraph checkpointer: saves graph state to a backend (SQLite, Redis) so agents can resume mid-task
  • Semantic memory: store facts as embeddings, retrieve by similarity using vector search (FAISS, Chroma)
  • Episodic memory: store past agent runs/interactions, let the agent learn from history
  • Memory summarization: when context gets too long, summarize old turns instead of dropping them

🛠️ Add SQLite checkpointing for cross-session memory

Make an agent remember facts from a previous conversation using LangGraph's SQLite checkpointer.

  1. Install: pip install langgraph-checkpoint-sqlite
  2. Create SqliteSaver pointing to a file: memory.db
  3. Compile graph with checkpointer=memory
  4. Run a conversation with thread_id="user-123" in the config
  5. Close Python, reopen, reload the graph — ask "What did I tell you earlier?" — it should remember!
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI

memory = SqliteSaver.from_conn_string("memory.db")
agent = create_react_agent(ChatOpenAI(), tools=[], checkpointer=memory)

config = {"configurable": {"thread_id": "session-1"}}

# Session 1: tell the agent something
agent.invoke({"messages": [("user", "My name is Ravi and I love Python")]}, config)

# Session 2 (new run): agent still remembers
result = agent.invoke({"messages": [("user", "What is my name?")]}, config)
print(result["messages"][-1].content)  # → "Your name is Ravi"

1.6
Logic Trees, Prompt Chains & Stateful Orchestration
LCEL chaining · StateGraph · Reducers · Multi-step reasoning
Exam Topic StateGraph · LCEL AgentIQ pipelines
▼
📖 What to Study
  • Prompt chaining: output of one LLM call feeds as input to the next — like a pipeline of prompts
  • Logic trees: decision points in a workflow where the path depends on model output
  • LangChain LCEL: pipe operator (|) to compose chains: prompt | llm | parser
  • LangGraph StateGraph: more powerful than LCEL for loops, branches, and persistent state across many steps
  • State reducers: when multiple nodes update the same state field, a reducer function merges them (e.g., operator.add for lists)
  • When to use LCEL vs StateGraph: LCEL for simple linear chains; StateGraph for complex, stateful, looping workflows

Integrate Knowledge Graphs for Relational Reasoning
Nodes · Edges · Graph querying · LLM + KG agent tool
Exam Topic NetworkX · Neo4j NeMo Retriever
▼
📖 What to Study
  • Knowledge graph (KG): a graph of entities (nodes) connected by relationships (edges) — e.g., Person → works_at → Company
  • Why KG for agents? Enables multi-hop reasoning: "Who manages the team that built product X?"
  • NetworkX: lightweight Python library for building and querying graphs in-memory
  • Neo4j: production graph database; uses Cypher query language
  • GraphRAG: instead of chunking documents, extract entities and relationships into a KG, then retrieve by traversing the graph
  • LangChain GraphCypherQAChain: lets an LLM write Cypher queries against Neo4j to answer questions

🛠️ Build a KG query tool and attach it to a LangGraph agent

Create a small knowledge graph (people, companies, roles) with NetworkX. Wrap a graph query as a @tool and attach it to a ReAct agent.

  1. Build a nx.DiGraph with 5+ people, their employers, and roles
  2. Write a @tool function that accepts a name and returns their relationships
  3. Add this tool to a create_react_agent
  4. Ask: "Who does Ravi work for, and what team does that company have?" — test multi-hop reasoning
  5. Bonus: replace NetworkX with Neo4j + GraphCypherQAChain
import networkx as nx
from langchain_core.tools import tool

# Build a small knowledge graph
G = nx.DiGraph()
G.add_edge("Ravi", "NVIDIA", relation="works_at")
G.add_edge("NVIDIA", "AI Team", relation="has_team")
G.add_edge("Priya", "NVIDIA", relation="works_at")

@tool
def query_knowledge_graph(entity: str) -> str:
    """Look up relationships for an entity in the knowledge graph."""
    if entity not in G:
        return f"Entity '{entity}' not found"
    edges = [(u, d['relation'], v) for u, v, d in G.edges(entity, data=True)]
    return str(edges)

🟣 LangGraph Activity
🛠️ Make your agent model-agnostic with fallbacks

Refactor an existing agent so the LLM is loaded from a config file, with a fallback chain if the primary model fails.

  1. Load LLM from a config.yml: provider: openai or provider: nvidia
  2. Create two LLM instances: primary_llm and fallback_llm
  3. Chain them: llm = primary_llm.with_fallbacks([fallback_llm])
  4. Test by setting a bad API key for primary — confirm fallback kicks in
  5. Convert agent to async: change invoke → ainvoke, wrap in asyncio.run()
import asyncio
from langchain_openai import ChatOpenAI
from langchain_nvidia_ai_endpoints import ChatNVIDIA

# Primary: OpenAI | Fallback: NVIDIA NIM
primary = ChatOpenAI(model="gpt-4o-mini")
fallback = ChatNVIDIA(model="meta/llama-3.1-8b-instruct")

llm = primary.with_fallbacks([fallback])

# Run 5 tasks in parallel (async)
async def run_parallel():
    tasks = [llm.ainvoke(f"Summarize topic {i}") for i in range(5)]
    results = await asyncio.gather(*tasks)
    return results

asyncio.run(run_parallel())
